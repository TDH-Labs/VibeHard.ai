  TRUST & SECURITY

Eleven gates. Every one shown.

In 2025, security researchers found more than 170 apps built on vibe-coding platforms leaking customer data — not from a sophisticated attack, but because the table holding it had no access rule at all. Below is exactly what stands between an AI-written app and that outcome here.

The failure mode this product exists to close

CVE-2025-48757. A missing or overly permissive row-level-security policy let an anonymous request read another customer's records. It's the single most common way an AI-generated app leaks — and it's exactly what Gate 4 and Gate 6, below, are built to catch before anything ships.

THE ELEVEN GATES, IN ORDER

The rule behind all of it: the AI proposes, the gate disposes.

The model writes the code. It does not decide whether the code is safe — eleven independent, deterministic checks do, and they fail closed: a gate that can't run counts as a block, never a silent pass.

GATE
01/11

sast — static code analysis

Reads every line of authored source for the patterns that become breaches: injection sinks, dangerous constructs, and a purpose-built SQL-injection rule layered on top of the standard scanner ruleset. It reads the code the way an attacker would, before anyone else gets the chance.

MECHANISM · semgrep, pinned container   ON FAIL · BLOCK
GATE
02/11

secrets — committed credentials

Hunts the full build for committed credentials — API keys, tokens, connection strings, passwords. Any leak blocks, regardless of severity; there's no judgment call here, a secret in code is a breach running on a timer.

MECHANISM · gitleaks, pinned container   ON FAIL · BLOCK
GATE
03/11

depvuln — dependency vulnerabilities

Every dependency in the lockfile is checked against the public CVE record. The fail-closed rule is literal: if the scanner itself can't run, that's a critical block on its own — a scan that didn't happen is never treated as a clean one.

MECHANISM · trivy, pinned container   ON FAIL · BLOCK · scanner down = BLOCK
GATE
04/11

rls — access-policy presence

Modeled directly on CVE-2025-48757. Every table a migration creates must carry row-level security and a policy that isn't a rubber stamp; every table the app actually queries must be covered. No policy found means block — there's no benefit of the doubt.

MECHANISM · static policy + coverage analysis   ON FAIL · BLOCK
GATE
05/11

migrate — migrations, executed

Every other check reads database migrations as text. This one runs them, against a real embedded Postgres, in about a second, with no LLM in the loop. Syntax that would only explode on deploy day explodes here instead, localized to the exact file and the exact database error.

MECHANISM · embedded Postgres, real DDL apply   ON FAIL · BLOCK
GATE
06/11

rls-enforce — the live attack

The keystone, and the reason this list exists. It stands up a live database, applies the real migrations, and seeds two tenants. Then, as tenant A and as an anonymous session, it attempts to read, update, delete, and insert tenant B's rows across every isolated table. One leaked row blocks the build. Full breakdown below.

MECHANISM · live two-tenant probe, embedded Postgres   ON FAIL · BLOCK · harness error = BLOCK
GATE
07/11

compliance — controls for sensitive data

Reads the binding sensitivity classification the intake interview produced and demands the controls that class of data requires — authentication in front of it, a real hard-delete path — routing the org-level judgment items to a person instead of hand-waving them. The classification can't be talked out of: declare "no sensitive data" while the schema stores diagnoses, and an independent code scan falsifies the claim and blocks the build.

MECHANISM · seven-control assessment, classification-driven   ON FAIL · BLOCK
GATE
08/11

pii — personal data in logs and URLs

Two deterministic leak paths: personal data written into logs (logs routinely ship to third parties), and personal data read from a URL or query string (recorded by every server in between). Precision matters — console.log("email sent") passes; console.log(user.email) blocks.

MECHANISM · property-access scan   ON FAIL · BLOCK
GATE
09/11

prod-readiness — operational hygiene

The boring failures that kill real handoffs: dependency ranges left unpinned so a silent upgrade breaks a shipped app, a missing README, containers running as root, base images not pinned by digest, type-checking turned off. Blocks at production rigor, warns at prototype — the standard scales with what the spec actually declared.

MECHANISM · install-free file checks, adaptive rigor   ON FAIL · BLOCK at production
GATE
10/11

verify — boot proof

The app is copied to a clean room, built for real, and booted — multiple times, and every single run has to come up healthy. One green run proves nothing; an empty build that "succeeded" without producing artifacts is also a block.

MECHANISM · clean-room build, repeated boot probes   ON FAIL · BLOCK
GATE
11/11

completeness — the promised features exist

An app can pass every security gate and still be missing half of what was asked for. Feature by feature, the code is held against the product spec's contract list. A promised feature that's entirely missing blocks the release, and the next fix round generates it. If the reviewer itself goes down, that's a block too: unverifiable is unshippable.

MECHANISM · per-feature review vs. spec contract   ON FAIL · BLOCK
EXIT

the sentinel — the deploy's one key

Once every gate passes, with at least one doing substantive work (eleven "not applicable" verdicts don't count as a pass), a cryptographically signed sentinel gets written, bound to that exact build in that exact directory. The deploy won't run without it, a copy of it elsewhere won't validate, and it's deleted the moment a later run fails.

MECHANISM · HMAC-signed, path-bound   WITHOUT IT · NO DEPLOY

WHY GATE 6 IS THE ONE THAT MATTERS

Reading a policy is an opinion. Attacking a database is a fact.

An AI can write access-control code that looks correct and still leaks — a typo'd condition, a policy that checks the wrong column, a table someone forgot to cover. Reading that code and concluding it's fine is exactly the failure mode behind CVE-2025-48757; the code looked fine to whoever shipped it, too.

So Gate 6 doesn't read and hope. It stands up a real Postgres database, applies the actual migrations the build produced, and seeds two separate tenants with real rows. Then, authenticated as tenant A and again as a completely anonymous session, it tries to read, write, update, and delete tenant B's data across every table the spec marked as isolated. The only proof that survives is the attack that fails. A single row coming back blocks the build immediately.

No badges. Receipts.

We will never hang a compliance badge we didn't earn — the compliance gate above is forbidden, by written policy, from ever claiming certification. What you get instead is the full verdict board: every gate's result, every finding, and a cryptographically signed release sentinel tied to the exact build. If a claim on this site ever drifts from what the gates actually do, the gates win.
