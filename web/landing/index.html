02 / THE UNCOMPROMISING SPEC

      
BLUEPRINT DECK · .vibehard/

    

    
The chassis is drawn before the metal.

    
Your sentence becomes structure first. Everything built afterward has to bolt to it.


    

      

        

          B-01 / INTAKE → SPECIFICATION
          
The questions come before the code.

          
Most generators sprint from your words straight to code. VibeHard refuses. First, a short intake interview extracts what your sentence implied but never said: who uses this, what data it holds, which of it is sensitive, and what must never cross between clients.

          
The answers are written into .vibehard/spec.json — including a formal sensitivity classification (PII, financial, health, credentials) and a rigor level. That file is load-bearing: every verdict later in the line is measured against it.

        

        

          B-02 / PRODUCT REQUIREMENTS
          
Every feature becomes a contract.

          
The PRD is a numbered set of obligations. Each feature written here is one the machinery will later be held to, item by item, by a gate whose only job is to answer: does the code actually do this?

          
A build that quietly ships four of your six features doesn't get to call itself finished. The paper remembers.

        

        

          B-03 / ARCHITECTURE
          
Isolation is drawn into the frame.

          
The architecture document decides the stack, the data model, and — critically — which tables belong to a single tenant and which are shared. Row-level security posture is settled here, at the drawing stage, where it's structural.

          
Access control added as an afterthought is how app builders end up in breach headlines. Here it's a property of the frame itself.

        

        

          B-04 / WHY THIS ISN'T PAPERWORK
          
The spec is the standard of proof.

          
A gate can only refuse a build if there's a written standard to refuse it against. The spec is that standard — which is why it can't be gamed. Declare "no sensitive data" while your schema stores diagnoses and card numbers, and an independent code scan falsifies the claim and blocks the build for lying.

          
From here, the drawings hand off to the machinery. The matrix takes over ↓

03 / THE GATE MATRIX

      
INSPECTION FLOOR · FIXED ORDER · FAIL-CLOSED

    

    
Eleven gates. Zero appeals.

    
Every build the factory produces is treated as contraband until proven otherwise. Eleven independent inspections run in fixed order, each with unilateral authority to stop the line, indifferent to your launch date. A gate that crashes mid-run still counts as a block.


    


    

      
Readout

      

        

        

      

    


    

      

        
GATE ▸

        
BLOCK ▸

        
FIX ▸

        
RE-GATE ▸

        
CONVERGE ▮ OR HOLD

      

      

        

          
A block is a work order

          Findings come back with file, line, and rule. The fixer rewrites and the line runs again — the fast chain every round, the full chain once at convergence against the real, clean-room artifact. The fixer holds no authority. Only the gate disposes.
        

        

          
The fixer can't cheat

          Before the fixer touches anything, the surface it must satisfy is captured. Weaken a policy, delete a check, switch off row-level security to make a finding "go away" — the tamper is detected and the round is voided. The bar stays where the spec set it.
        

        

          
Rounds are bounded

          No infinite thrash. If the loop exhausts its rounds, the build is held — packaged with every verdict and finding and routed to a human. It does not ship broken. There is no flag to make it ship broken.
